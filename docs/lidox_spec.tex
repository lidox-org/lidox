\section{Functional Requirements}

\begin{decisionbox}[Format Convention]
\small Each requirement specifies: unique ID, description, triggering condition, expected system behavior, and acceptance criteria---per the assignment rubric.
\end{decisionbox}

\subsection{Area 1: Real-Time Collaboration}

{\small\textit{High-level capability:} Multiple users simultaneously edit the same document with sub-second state synchronization, live presence awareness, and deterministic conflict resolution without data loss.}

\frcard{FR-1.1}{State Synchronization (CRDT)}{The system shall synchronize document state across concurrent clients using a Conflict-free Replicated Data Type (CRDT) transmitted via persistent WebSocket connections (RFC~6455).}{A user types, deletes, or formats text in a shared document.}{The CRDT engine generates a local operation, applies it optimistically to the local editor state, and broadcasts the operation to all connected peers via the Real-Time Service. All replicas converge to an identical document state without coordination.}{Given 2+ concurrent editors making simultaneous edits to the same paragraph, the final document state is identical across all clients within 300\,ms (p99) and no characters are lost.}

\frcard{FR-1.2}{Presence Broadcasting}{The system shall display real-time presence indicators (cursor position, active selection, user name, avatar color) for all connected collaborators.}{A user moves their cursor or changes their text selection.}{The client sends a lightweight presence update to the Real-Time Service via WebSocket. The service fans out the update to all other connected clients of that document via Redis Pub/Sub. Presence data is ephemeral (never written to the primary database).}{When User~A moves their cursor, User~B sees the updated cursor position within 200\,ms. When User~A disconnects, their cursor disappears within 60\,s (TTL expiry).}

\frcard{FR-1.3}{Offline Reconciliation}{The client shall buffer local edits during network partitions and merge them deterministically upon reconnection.}{The client detects a WebSocket disconnection (network loss, server restart).}{Local CRDT operations are buffered in the browser's IndexedDB. Upon reconnection, the client sends buffered operations to the server and receives any missed remote operations. The CRDT merge algorithm resolves all conflicts deterministically without user intervention. If offline for $>$24\,h, the client receives a full compacted snapshot instead of incremental operations.}{A user edits 3 paragraphs while offline for 10 minutes. Upon reconnection, all offline edits appear in the document within 5\,s and no data from either the offline or online users is lost.}

\subsection{Area 2: AI Writing Assistant}

{\small\textit{High-level capability:} Users invoke contextual AI operations (rewrite, summarize, translate, restructure) on selected text. AI suggestions are presented as reviewable proposals anchored to CRDT state, with full format preservation and task-aware cost management.}

\frcard{FR-2.1}{AI Invocation}{The user can invoke an AI operation on selected text via a contextual menu.}{The user selects text and chooses an AI action (rewrite, summarize, translate, restructure, grammar fix) from the toolbar or keyboard shortcut.}{The frontend captures the selected text, the CRDT node ID, and the current state vector. It sends the request to the Backend API, which routes it to the AI Orchestration Service. The AI service constructs a prompt using a task-aware token budget, enqueues the job, and returns an immediate acknowledgment with a task ID. The client shows a pending/loading state.}{The AI pending indicator appears within 300\,ms of invocation. The first token of the streamed response appears within 2\,s for selections under 1,000 words.}

\frcard{FR-2.2}{AI Proposal Display (Diff UX)}{AI-generated text shall render as a reviewable diff proposal, not as an automatic replacement.}{The AI Orchestration Service completes processing and returns the generated text.}{The client renders the AI output as a tracked-change-style overlay: additions highlighted in green, deletions in red. The user can accept all, reject all, accept individual sentences, or manually edit the proposed text before committing. Proposals are anchored to CRDT node IDs (not character offsets) and include a conflict resolution mechanism: if the source text has been modified by a collaborator during AI processing, the proposal is marked ``stale'' and the user is prompted to regenerate.}{(1) The user can accept/reject an AI proposal. (2) Partial acceptance (individual sentences) works correctly. (3) A proposal generated against text that was concurrently edited by another user displays a ``stale'' indicator rather than silently applying.}

\frcard{FR-2.3}{Rich-Text Preservation}{The AI pipeline shall preserve formatting (bold, italic, links, headings, lists, tables) through the prompt--response round trip.}{The user invokes an AI operation on formatted text.}{The ProseMirror-to-LLM serializer converts document nodes to a compact structured format with inline semantic tags. The AI response parser reconstructs ProseMirror nodes from the output, rejecting any output that violates the document schema. Tables are serialized as CSV-like representations; code blocks are passed verbatim.}{A paragraph containing bold text, a hyperlink, and an inline code span, when rewritten by the AI, retains all three formatting marks in the output.}

\frcard{FR-2.4}{AI Restructure / Outline}{The user can ask the AI to restructure a document outline or reorder sections.}{The user selects multiple headings/sections and invokes ``Restructure.''}{The AI receives the full heading tree and section summaries (heading-bounded chunking). It returns a proposed new outline with reordered/renamed sections. The proposal shows a before/after outline diff. Acceptance rearranges the actual document nodes.}{Given a 5-section document, the AI produces a restructured outline. The user can preview the new order and accept or reject it. Upon acceptance, all content moves to the correct positions.}

\subsection{Area 3: Document Management}

{\small\textit{High-level capability:} Full document lifecycle management including creation, versioning, sharing with role-based permissions, and export to standard formats.}

\frcard{FR-3.1}{Document Creation}{Users can create new documents with optional templates.}{The user clicks ``New Document'' from the dashboard.}{The system creates a new document record in PostgreSQL, initializes an empty CRDT state in Redis, assigns the creating user as Owner, and redirects to the editor view. The new document is immediately editable.}{A newly created document appears in the user's dashboard within 1\,s and is editable by the creator immediately.}

\frcard{FR-3.2}{Version History \& Revert}{The system shall maintain document version history and allow authorized users to revert to any previous version.}{The user opens the version history panel or clicks ``Revert'' on a specific version.}{The system persists snapshots at configurable intervals (every 100 CRDT operations, every 5 minutes, or on explicit save). The version history panel shows timestamped snapshots with author attribution. Reverting replaces the current CRDT state with the snapshot state and broadcasts the change to all connected clients.}{(1) At least 10 historical versions are available for a document edited over 1 hour. (2) Reverting to a version from 30 minutes ago replaces the content for all connected users within 3\,s.}

\frcard{FR-3.3}{Document Sharing \& Access Control}{Users can share documents with specific users or via link, with role-based permissions.}{The document owner opens the sharing panel and enters an email address or generates a share link with a specific role.}{The system creates a permission record linking the target user/link to the document with one of four roles: Owner, Editor, Commenter, or Viewer. Permissions are enforced on every API request and WebSocket message. Commenter role allows AI ``Analyze'' but disables ``Apply Changes.'' Viewer role disables all editing and AI invocation.}{(1) A user shared as ``Viewer'' cannot type in the editor. (2) A ``Commenter'' can invoke AI analysis but the ``Apply'' button is disabled. (3) Permission changes take effect within 60\,s for active sessions.}

\frcard{FR-3.4}{Document Export}{The system shall export documents to PDF and DOCX formats.}{The user clicks ``Export'' and selects a format.}{The backend renders the current document state to the requested format. Unaccepted AI suggestions are rendered as margin annotations (tracked changes in DOCX, comment boxes in PDF). The exported file is returned as a downloadable binary.}{An exported DOCX file opens correctly in Microsoft Word with formatting intact. Unresolved AI proposals appear as tracked changes.}

\subsection{Area 4: User Management}

{\small\textit{High-level capability:} Secure authentication with federated SSO, short-lived tokens with active revocation, and role-based authorization enforced across all system boundaries.}

\frcard{FR-4.1}{Authentication (Local + SSO)}{The system shall support email/password authentication and SSO via Google and GitHub.}{The user navigates to the login page and enters credentials or clicks an SSO button.}{Local credentials are validated against bcrypt-hashed passwords (cost factor $\ge$12). SSO follows the OAuth~2.0 Authorization Code flow (RFC~6749) with PKCE. Upon successful authentication, the server issues an access JWT (15-min TTL, HttpOnly Secure cookie) and an opaque refresh token (7-day TTL, stored server-side). The user is redirected to their dashboard.}{(1) A user can log in with email/password. (2) A user can log in with Google SSO. (3) The JWT cookie has \texttt{HttpOnly}, \texttt{Secure}, and \texttt{SameSite=Strict} flags.}

\frcard{FR-4.2}{Session Management \& Token Revocation}{The system shall enforce short-lived sessions with active revocation capability.}{An admin revokes a user's access, or a user logs out, or a refresh token is reused (replay attack detected).}{The revoked JWT's \texttt{jti} is added to a Redis deny set (\texttt{SISMEMBER} check on every request). A Pub/Sub revocation event forces all Real-Time Service nodes to terminate the affected WebSocket connections within 5\,s. Refresh token rotation invalidates old tokens; reuse of an old token triggers revocation of the entire token family.}{(1) After revocation, the user's API requests return 401 within 15 minutes (JWT expiry). (2) The user's WebSocket connection is terminated within 60\,s (re-auth ping cycle). (3) Refresh token reuse triggers immediate session termination.}

\frcard{FR-4.3}{Role Enforcement}{The system shall enforce four document-level roles: Owner, Editor, Commenter, Viewer.}{Any API request or WebSocket message that modifies document state or invokes AI.}{The Backend API and Real-Time Service check the requesting user's role against the required permission level for the action. Unauthorized actions return HTTP~403 or a WebSocket error frame. The AI Orchestration Service additionally checks whether the org admin has disabled AI for the document.}{(1) A Viewer attempting to send a CRDT operation receives a 403 error. (2) A Commenter invoking ``AI Rewrite'' receives a 403; invoking ``AI Analyze'' succeeds. (3) An Org Admin disabling AI for a document blocks all AI invocations for all roles.}

\section{Non-Functional Requirements}

\subsection{Latency}
\begin{speclist}
\item \requirement{NFR-01} Keystroke propagation to connected peers: p50 $\le$80\,ms, p95 $\le$150\,ms, p99 $\le$300\,ms. \textit{Justification:} Research shows collaborative presence breaks at $>$200\,ms; $\le$150\,ms (p95) maintains the psychological illusion of instantaneous shared editing.
\item \requirement{NFR-02} AI pending indicator: $\le$300\,ms. First token streaming: $\le$2\,s for tasks under 1,000 words. \textit{Justification:} $>$300\,ms without feedback causes users to re-click; $>$3\,s feels ``broken.''
\item \requirement{NFR-03} Document load (up to 100 pages, broadband): $\le$1.5\,s to interactive. \textit{Justification:} Google's research shows 53\% of mobile users abandon sites that take $>$3\,s to load.
\end{speclist}

\subsection{Scalability}
\begin{speclist}
\item \requirement{NFR-04} 50 concurrent active editors per document without violating NFR-01.
\item \requirement{NFR-05} 10,000 active documents and 100,000 authenticated users system-wide. \textit{Growth model:} Initial target supports classroom and small-enterprise scenarios. Horizontal scaling via Redis Cluster partitioning (by document ID) and stateless backend pods.
\end{speclist}

\subsection{Availability}
\begin{speclist}
\item \requirement{NFR-06} Core editing platform: 99.9\% monthly uptime (8.76\,h/year max downtime).
\item \requirement{NFR-07} If the AI subsystem is unavailable, users can still create, edit, save, share, and export documents (graceful degradation). The AI toolbar shows a ``temporarily unavailable'' indicator.
\item \requirement{NFR-08} During temporary real-time service failure, clients buffer edits locally in IndexedDB and auto-sync on reconnection. Users see a ``reconnecting'' indicator but are never locked out of their document.
\end{speclist}

\subsection{Security \& Privacy}
\begin{speclist}
\item \requirement{NFR-09} All data in transit: TLS~1.3 (RFC~8446). All data at rest: AES-256 (ISO~27001 compliance).
\item \requirement{NFR-10} JWT access tokens: 15-min TTL. Active revocation latency: $\le$60\,s for WebSocket, $\le$15\,min for REST.
\item \requirement{NFR-11} AI context sends minimum required data per task type. Raw prompt/response retained $\le$30 days (org-configurable to 0). Document data never sent to LLM if org admin disables AI. \textit{Privacy implication:} Sending content to third-party LLM APIs means the LLM provider may process/log that content. Org admins must be able to opt out entirely for sensitive documents.
\end{speclist}

\subsection{Usability}
\begin{speclist}
\item \requirement{NFR-12} When $>$10 collaborators are active, the presence UI collapses inactive cursors into a summary indicator (``+8 others'') to prevent clutter.
\item \requirement{NFR-13} AI suggestions are always previewable proposals that can be accepted, rejected, or edited before insertion. The user is never surprised by automatic content changes.
\item \requirement{NFR-14} WCAG~2.1 Level~AA compliance. The editor is keyboard-navigable. Screen readers announce AI proposal state changes (``AI suggestion available,'' ``suggestion accepted'').
\end{speclist}

\section{User Stories and Scenarios}

\begin{notebox}[Story Format]
\small Each story includes expected system behavior and a design justification for non-obvious choices, per the rubric requirement.
\end{notebox}

\subsubsection*{Collaboration Scenarios}

\uscard{US-01}{Presence Awareness}{As a collaborator, I want to see labeled cursors of other active users so I avoid overwriting their work.}{When a user opens a shared document, the Real-Time Service sends the current presence state of all connected users. Each collaborator's cursor is rendered with their display name and a unique color. Cursor positions update at up to 10\,Hz (server-throttled). When a user disconnects, their cursor disappears after the 60\,s TTL expires.}{Presence is broadcast via Redis Pub/Sub independently of document state to avoid write amplification on the primary database. Throttling to 10\,Hz prevents fan-out amplification in documents with 50 editors.}

\uscard{US-02}{Offline Editing \& Reconnection}{As a commuter, I want to edit during a tunnel outage and have my changes seamlessly merge when I reconnect.}{The client detects disconnection and shows a ``Working offline'' banner. All local edits continue normally against the local CRDT state and are buffered in IndexedDB. Upon reconnection, buffered operations are sent to the server and remote operations are received. The CRDT merge algorithm resolves all conflicts deterministically. The ``offline'' banner disappears.}{We use CRDTs (not OT) specifically because they support offline-first merging without server round-trips. The 24-hour snapshot fallback prevents unbounded operation replay for long-offline clients.}

\uscard{US-03}{Concurrent Same-Paragraph Editing}{As a co-author, I want to edit a paragraph simultaneously with a colleague without locking.}{Both users' keystrokes are applied locally via optimistic UI and broadcast via CRDT operations. The CRDT algorithm (Yjs/YATA) ensures character-level interleaving without paragraph-level locks. Neither user experiences any blocking or conflict dialog.}{Paragraph-level locking was rejected (ADR-02) because it destroys the collaborative experience. CRDTs allow true character-level concurrency at the cost of higher implementation complexity.}

\uscard{US-04}{Version Revert During Active Editing}{As a team lead, I want to revert to a version from 2 hours ago, forcing a live refresh for all users.}{The user opens the version history panel, selects a snapshot, and clicks ``Revert.'' The server replaces the current CRDT state with the snapshot state, broadcasts the new state to all connected clients via WebSocket, and each client replaces its local editor state. All cursors reset to position 0. A version entry is created recording who reverted and when.}{A forced revert is disruptive by design---it is an emergency action. We broadcast the full replacement state rather than trying to compute a delta, because reverting to an old version may invalidate hundreds of intermediate CRDT operations.}

\subsubsection*{AI Assistant Workflows}

\uscard{US-05}{AI Summarization}{As a writer, I want to select a long section and ask the AI for a bulleted summary.}{The user selects text spanning multiple paragraphs and clicks ``Summarize'' from the AI toolbar. The system shows a loading indicator within 300\,ms. The AI Orchestration Service constructs a prompt using heading-bounded context (up to 8,000 tokens). The response streams back and renders as a diff proposal showing the original text and the proposed summary. The user can accept, reject, or edit before insertion.}{Summarization requires more context than grammar fixes. The task-aware token budget (8,000 for summarize vs.\ 800 for grammar) prevents hallucination from insufficient context while avoiding unnecessary cost for simple operations.}

\uscard{US-06}{Partial AI Acceptance}{As an editor, I want to accept only specific sentences from an AI rewrite while rejecting others.}{The diff proposal renders each sentence as an independently selectable unit. The user clicks checkboxes next to individual sentences to include or exclude them. Clicking ``Apply Selected'' commits only the checked sentences to the CRDT state. Unchecked sentences are discarded.}{Partial acceptance is critical for maintaining authorial voice. The alternative---all-or-nothing acceptance---forces users to manually re-edit AI output, negating the productivity benefit.}

\uscard{US-07}{AI Translation with Format Preservation}{As a researcher, I want to translate a section while preserving bolding and hyperlinks.}{The user selects formatted text and invokes ``Translate to [language].'' The structured serializer converts ProseMirror nodes to tagged format, preserving inline markers. The AI receives the tagged text with a system prompt instructing format preservation. The response parser reconstructs ProseMirror nodes with original formatting intact.}{We serialize to a compact tagged format (not Markdown) because Markdown introduces round-trip ambiguity (e.g., \texttt{*} vs.\ \texttt{\_} for italics). Tagged format tokenizes more predictably and preserves exact formatting markers.}

\uscard{US-08}{AI Suggestion Becomes Stale During Collaboration}{As a user, I request an AI rewrite. While waiting for the result, my colleague edits the same paragraph. The AI suggestion arrives but is now based on outdated text.}{The system detects that the source text's SHA-256 hash no longer matches the current node content. Instead of silently applying the stale proposal, the system shows it with a yellow ``Stale: text has changed'' banner and a ``Regenerate'' button. The user can choose to regenerate with updated text or dismiss the proposal.}{Silently applying stale proposals would corrupt content---a trust-destroying failure. We anchor proposals to CRDT state vectors (ADR-06) rather than character offsets specifically to detect and handle this race condition.}

\subsubsection*{Document Lifecycle}

\uscard{US-09}{Share with Read-Only Access}{As a team lead, I share a document with a client as ``Viewer.'' They can read but not edit, invoke AI, or comment.}{The owner enters the client's email in the sharing panel and selects ``Viewer.'' The system creates a permission record. When the client opens the document, the editor renders in read-only mode: the cursor is non-interactive, the toolbar is disabled, and the AI menu is hidden.}{We enforce permissions at both the UI level (disabled controls) and the API level (403 on write attempts) for defense-in-depth. UI-only enforcement is insufficient because a technical user could send raw API requests.}

\uscard{US-10}{Export with AI Annotations}{As a project manager, I export a document. Unresolved AI suggestions appear as tracked changes.}{The user clicks Export $\to$ DOCX. The backend iterates over any pending AI proposals and converts them to DOCX tracked-change markup (using \texttt{w:ins} and \texttt{w:del} elements). The resulting file opens in Microsoft Word with suggestions visible in the review pane.}{Discarding unresolved suggestions on export would lose work. Rendering them as tracked changes leverages an existing UX pattern that document reviewers already understand.}

\subsubsection*{Access Control \& Roles}

\uscard{US-11}{Commenter Invokes AI Analyze (Allowed) vs.\ AI Rewrite (Blocked)}{As a commenter, I can use AI to analyze text for my understanding, but I cannot apply changes to the document.}{The user with ``Commenter'' role selects text and opens the AI menu. ``Analyze'' and ``Explain'' options are available; ``Rewrite,'' ``Summarize,'' and ``Translate'' show a lock icon with tooltip ``Requires Editor role.'' If the commenter invokes ``Analyze,'' the AI response displays in a side panel (read-only) with no ``Apply to Document'' button.}{Commenters need AI for comprehension (analyzing complex text) but must not modify shared content. The role matrix differentiates between read-side AI (analysis) and write-side AI (rewrite/summarize). This granularity was requested by enterprise stakeholders.}

\uscard{US-12}{Org Admin Disables AI for Confidential Documents}{As an Org Admin, I disable AI features for a specific document to prevent sensitive content from being sent to external LLM APIs.}{The admin opens the document settings panel and toggles ``AI Features'' to off. The system sets an \texttt{ai\_enabled=false} flag on the document record. All AI invocations for this document are immediately blocked at the API layer (returning 403 with message ``AI disabled by administrator''). The AI toolbar in the editor is hidden for all users.}{Enterprise customers handling legal, medical, or financial data cannot risk any content reaching third-party LLM providers. This toggle provides document-level granularity rather than org-wide, allowing mixed environments.}

\uscard{US-13}{AI Interaction History Review}{As a professor, I want to review AI interaction history to understand how a section evolved and distinguish AI-generated from manually typed text.}{The version history panel includes an ``AI Activity'' tab showing: timestamp, user who invoked AI, task type (rewrite/summarize/translate), source text snippet, and whether the suggestion was accepted, rejected, or partially applied. AI-contributed text is subtly marked in the editor with a small indicator icon. The audit log retains this data for the org-configurable retention period.}{This supports academic integrity use cases and editorial review workflows. The retention policy balances auditability with GDPR Article~17 (right to erasure) obligations.}

\section{Architectural Drivers}

\begin{decisionbox}[Why This Ranking Matters]
\small Two teams with different driver rankings should arrive at different architectures. Our ranking reflects a product where \emph{real-time collaboration reliability} is non-negotiable, \emph{AI quality/cost} is the primary business risk, and \emph{security} is the enterprise sales gate.
\end{decisionbox}

\noindent\textbf{Rank 1: Real-Time Collaboration Latency \& Reliability} {\small(NFR-01, NFR-06, NFR-08)}\\[2pt]
{\small If sync is slow or lossy, the product is unusable regardless of AI quality. This driver forces: CRDT-based conflict resolution (ADR-02), a dedicated Real-Time Service separated from CRUD traffic (ADR-01), a tiered state architecture with Redis for ephemeral/warm data (ADR-05), and offline-first client design. Every infrastructure choice flows downstream from the $\le$150\,ms p95 latency target.}

\vspace{6pt}
\noindent\textbf{Rank 2: AI Integration Quality \& Cost Control} {\small(NFR-02, NFR-11, FR-2.1--2.4)}\\[2pt]
{\small The AI assistant is the product differentiator but also the primary cost and reliability risk. This driver forces: task-aware token budgeting with model tiering (ADR: implicit in FR-2.1), asynchronous queue-backed execution (ADR-04), anchor-based proposal conflict resolution (ADR-06), and per-org cost circuit breakers. Without these, a single viral document can generate thousands of dollars in LLM costs in hours.}

\vspace{6pt}
\noindent\textbf{Rank 3: Security \& Compliance} {\small(NFR-09, NFR-10, FR-4.1--4.3)}\\[2pt]
{\small Enterprise customers will not adopt a tool that fails security review. This driver forces: active JWT revocation with Redis deny set (ADR-07), document-level AI kill switch, TLS~1.3 everywhere, AES-256 at rest, and configurable data retention policies. The 60-second WebSocket revocation target specifically addresses ISO~27001 A.9.2.6.}

\vspace{6pt}
\noindent\textbf{Rank 4: Developer Velocity \& Maintainability} {\small(Stakeholder: Engineering Team)}\\[2pt]
{\small A startup must ship fast. This driver forces: monorepo with shared type definitions, API-first contracts between services, mockable AI orchestration layer, and clear code ownership boundaries. It also informs our choice of proven technologies (React, NestJS, Yjs) over novel but risky alternatives.}

\subsection{AI Integration Design}

\subsubsection{Context and Scope}
The AI sees \textbf{only the minimum context required} for the task, never the full document by default. A ``Grammar Fix'' operation receives the selected text plus one sentence before/after (800 token ceiling). A ``Summarize'' operation receives the full heading-bounded section (8,000 tokens). A ``Full Document Analysis'' is the only task that sees the complete document (chunked at 16,000 tokens per request). This minimizes cost, latency, and data exposure to the LLM provider.

For very long documents ($>$16K tokens), the system uses \textbf{heading-based chunking}: it splits the document at heading boundaries and processes each chunk separately, assembling the results on the client.

\subsubsection{Suggestion UX}
AI suggestions are presented as \textbf{tracked-change-style proposals}: additions highlighted green, deletions highlighted red, rendered inline in the editor. The user can accept all, reject all, accept individual sentences, or manually edit the proposal text. A side panel shows a clean ``before/after'' view. Proposals auto-dismiss after 120 seconds of inactivity to prevent ghost suggestions. Accepted suggestions are committed to the CRDT state as a single atomic operation; undo reverts the entire acceptance.

\subsubsection{AI During Collaboration}
We do \textbf{not} lock the region during AI processing. Other users can continue editing freely. When the AI result arrives, the anchor-based conflict resolution system (FR-2.2) detects whether the source text has changed. If unchanged: apply directly. If surrounding text changed: rebase offsets and show a ``Context Changed'' indicator. If the source text itself changed: mark as stale, prompt ``Regenerate?'' This was chosen over region-locking because locking a paragraph for 2--8 seconds in a multi-user session is unacceptably disruptive.

\subsubsection{Prompt Design}
Prompts are \textbf{template-based}, stored as YAML files in the repository under \texttt{packages/ai-service/prompts/}. Each template has a system prompt (persona, constraints), a user prompt (dynamic: selected text, context), and optional few-shot examples. Templates are hot-reloadable: changing a YAML file and restarting the AI service updates prompts without a full application redeployment. Future iteration: configurable prompts per organization.

\subsubsection{Model and Cost Strategy}
We use \textbf{tiered model routing}: fast models (e.g., Claude Haiku) for grammar/spelling (high frequency, low complexity), standard models (e.g., Claude Sonnet) for rewrite/translate/summarize, and premium models (e.g., Claude Opus) for full-document analysis. This reduces LLM API costs by an estimated 40--60\% versus a single-model approach. Cost management: per-org monthly budget with 80\% alert and 100\% hard-stop. Per-request ceiling of \$0.50 requires explicit user confirmation. Token counts are metered per request and surfaced in an admin dashboard.

}

\textbf{Privacy implications of LLM APIs:} Sending content to third-party LLM providers means the provider may process, log, or temporarily store that content. Mitigations: (1)~send minimum context per task; (2)~org admins can disable AI per document; (3)~AI interaction logs are retained for $\le$30 days (configurable to 0); (4)~system prompt instructs the LLM not to retain or learn from user content.

\subsection{Communication Model}

\textbf{Model:} Push-based real-time communication via persistent WebSocket connections. When a user edits the document, the CRDT operation is applied locally (optimistic UI) and sent to the Real-Time Service over WebSocket. The service relays the operation to all other connected clients. There is no polling.

\textbf{Trade-offs:} WebSockets provide the lowest-latency bidirectional communication (critical for our $\le$150\,ms p95 target) but require persistent connections, which are more complex to scale horizontally than stateless HTTP. We accept this complexity because polling-based sync (Server-Sent Events + HTTP POST) would add 200--500\,ms of latency per keystroke, destroying the collaboration experience.

\textbf{Document open:} When a user opens a shared document, the client (1)~fetches the initial document metadata via REST, (2)~upgrades to a WebSocket connection, (3)~receives the current CRDT state snapshot and presence state, (4)~renders the document and begins accepting edits.

\textbf{Disconnection and reconnection:} On WebSocket disconnection, the client switches to offline mode: edits are applied locally and buffered in IndexedDB. A ``reconnecting'' banner is shown. On reconnection, buffered operations are replayed to the server, and missed remote operations are received. CRDT merge ensures consistency. If offline for $>$24\,h, the client receives a full compacted snapshot.

\subsection{Versioning Strategy}
Documents are not stored as monolithic text blobs. The \emph{live} document state is the CRDT state held in Redis (Tier~2), which is asynchronously persisted to PostgreSQL at $\le$5\,s intervals as a binary CRDT snapshot. Named \emph{version snapshots} are triggered by: every 100 CRDT operations, every 5 minutes of active editing, or explicit user action. Snapshots are stored in Object Storage (S3) and referenced by URL in the \texttt{document\_versions} table.

\subsection{AI Interaction History}
Every AI invocation creates a row in \texttt{ai\_interactions} recording the full audit trail: who invoked what, on which document, with what cost, and whether the result was accepted. This supports the professor use case (US-13) and org admin auditing. Raw prompt/response text is stored in a separate \texttt{ai\_interaction\_details} table with TTL-based auto-deletion per the org's \texttt{ai\_retention\_days} setting.

\section{Architecture Decision Records}

\begin{adrcard}[ADR-01: Dedicated Real-Time Collaboration Service \hfill \statusAccepted]
\small
\begin{tabularx}{\textwidth}{@{}>{\bfseries\sffamily\color{inkMuted}}l>{\raggedright\arraybackslash}X@{}}
Context & Real-time editing has fundamentally different latency requirements than standard CRUD. WebSocket connections are long-lived and stateful; REST endpoints are short-lived. Mixing them in one service couples their scaling and failure modes. \\
Decision & Separate real-time synchronization into a dedicated service (C3) with its own scaling group. REST API (C2) is a separate service. \\
Consequences & \textbf{+} Performance-critical traffic is isolated; each service scales independently. \textbf{--} Inter-service coordination adds complexity; document state must be accessible from both services. \\
Alternatives & \emph{Single monolith}---simpler but latency-coupled. \emph{Polling-based sync}---inferior UX, higher server load. \\
\end{tabularx}
\end{adrcard}

\begin{adrcard}[ADR-02: CRDT-Based Conflict Resolution (Yjs/YATA) \hfill \statusAccepted]
\small
\begin{tabularx}{\textwidth}{@{}>{\bfseries\sffamily\color{inkMuted}}l>{\raggedright\arraybackslash}X@{}}
Context & The platform requires simultaneous multi-user editing without locks. The offline reconciliation requirement (FR-1.3) demands merge without server round-trips. \\
Decision & Use Yjs (YATA algorithm) for its production maturity, compact binary encoding, TipTap integration, and native GC hooks. \\
Consequences & \textbf{+} True concurrent editing without locks; offline-first by design. \textbf{--} Higher complexity; tombstone accumulation requires GC (NFR-07 defines triggers). \\
Alternatives & \emph{OT}---requires central server for transformation, conflicts with offline-first. \emph{Pessimistic locking}---destroys collaboration UX. \emph{Last-write-wins}---causes data loss. \\
\end{tabularx}
\end{adrcard}

\begin{adrcard}[ADR-03: AI Outputs as Reviewable Proposals (Not Auto-Replace) \hfill \statusAccepted]
\small
\begin{tabularx}{\textwidth}{@{}>{\bfseries\sffamily\color{inkMuted}}l>{\raggedright\arraybackslash}X@{}}
Context & AI rewrites can be incorrect or tonally inappropriate. Auto-replacing content without consent is trust-destroying, especially in shared documents. \\
Decision & All AI outputs render as tracked-change-style diff proposals requiring explicit user action (accept/reject/partial/edit) before CRDT commit. Proposals auto-dismiss after 120\,s. \\
Consequences & \textbf{+} User maintains complete control; partial acceptance enables fine-grained editing. \textbf{--} Extra frontend complexity; users must take an additional action. \\
Alternatives & \emph{Inline auto-replacement}---fast but risky in shared docs. \emph{Full-document auto-rewrite}---destructive as default. \\
\end{tabularx}
\end{adrcard}

\begin{adrcard}[ADR-04: Async Queue-Backed AI Execution \hfill \statusAccepted]
\small
\begin{tabularx}{\textwidth}{@{}>{\bfseries\sffamily\color{inkMuted}}l>{\raggedright\arraybackslash}X@{}}
Context & LLM responses range from 2--15\,s with unpredictable latency spikes and rate limits. Blocking threads on LLM calls would exhaust the connection pool. \\
Decision & Route AI requests through a durable job queue (BullMQ on Redis). Client receives immediate \texttt{202} with task ID; results stream via WebSocket. Retries use exponential backoff with jitter. \\
Consequences & \textbf{+} Core app never blocks on LLM latency; clean retry semantics; quota enforcement at queue level. \textbf{--} More infrastructure; client handles async status updates. \\
Alternatives & \emph{Synchronous backend-to-LLM}---fragile. \emph{Direct client-to-LLM}---exposes API keys, bypasses audit. \\
\end{tabularx}
\end{adrcard}

\section{Development Workflow}

\subsection{Branching Strategy}

We use \textbf{GitHub Flow} (trunk-based with feature branches):

\begin{speclist}
\item \textbf{Main branch (\texttt{main}):} Always deployable. Protected: requires passing CI and at least 1 approved review.
\item \textbf{Feature branches:} Named \texttt{feat/\{issue-number\}-short-description} (e.g., \texttt{feat/42-ai-proposal-overlay}).
\item \textbf{Fix branches:} Named \texttt{fix/\{issue-number\}-short-description}.
\item \textbf{Merge policy:} Squash merge to \texttt{main}. The squash commit message follows the Angular Conventional Commit format: \texttt{type(scope): description} (e.g., \texttt{feat(ai): add task-aware token budgeting}).
\item \textbf{No long-lived branches:} Feature branches should be merged within 3 days. If a feature takes longer, break it into smaller PRs behind a feature flag.
\end{speclist}

\subsection{Code Review Process}

\begin{speclist}
\item \textbf{Every PR requires at least 1 approval} from a team member who is \emph{not} the author.
\item \textbf{Cross-area PRs} (touching $>$1 ownership area) require approval from each affected area owner.
\item \textbf{Review criteria:} (1) Does it match the API contract in \texttt{types}? (2) Are there tests? (3) No secrets in code. (4) No \texttt{console.log} in production code. (5) TypeScript strict mode passes.
\item \textbf{Review SLA:} Reviews should be completed within 24 hours. If not, the author pings in the team channel.
\item \textbf{CI gates:} Linting (ESLint), type checking (TypeScript), unit tests (Jest), integration tests (Supertest) must all pass before merge.
\end{speclist}

\subsection{Issue Tracking \& Task Assignment}

\begin{speclist}
\item \textbf{Tool:} GitHub Issues with project board (Kanban: Backlog $\to$ In Progress $\to$ In Review $\to$ Done).
\item \textbf{Task granularity:} Each issue represents 1--3 days of work. Larger features are broken into sub-issues.
\item \textbf{Assignment:} Issues are self-assigned during weekly planning. The ownership areas guide who picks what, but anyone can pick cross-cutting tasks.
\item \textbf{Labels:} \texttt{frontend}, \texttt{backend}, \texttt{ai}, \texttt{infra}, \texttt{cross-cutting}, \texttt{bug}, \texttt{blocked}.
\end{speclist}

\subsection{Communication \& Decision Documentation}

\begin{speclist}
\item \textbf{Daily:} Async standup in a dedicated Discord/Slack channel (what I did, what I'm doing, blockers).
\item \textbf{Weekly:} 30-minute synchronous planning meeting to review the board and assign the next sprint's issues.
\item \textbf{Decisions:} Any architectural or technical decision is documented as an ADR in \texttt{docs/decisions/} or as a comment on the relevant GitHub issue. Chat messages are ephemeral; decisions must live in the repo.
\end{speclist}

\section{Development Methodology}

\subsection{Methodology: Scrum-Inspired Sprints (Modified)}

We use a \textbf{lightweight Scrum framework} adapted for a 4-person team:

\begin{speclist}
\item \textbf{Sprint length:} 1 week. Short sprints force frequent integration and early detection of issues.
\item \textbf{Sprint planning:} Monday, 30 minutes. Review backlog, assign issues for the week.
\item \textbf{Sprint review:} Friday, 15 minutes. Demo what was completed; update stakeholders.
\item \textbf{No daily standup meetings:} Replaced by async text standups (see above). A 4-person team doesn't need synchronous daily ceremonies.
\end{speclist}

\subsection{Backlog Prioritization}

The backlog is prioritized using a \textbf{MoSCoW} framework aligned with architectural drivers:

\begin{specenum}
\item \textbf{Must Have (Sprint 1--2):} Core editing + real-time sync + basic auth + document CRUD. These are the architectural skeleton; nothing else works without them.
\item \textbf{Must Have (Sprint 3--4):} AI invocation pipeline + proposal UX + role enforcement. The product differentiator.
\item \textbf{Should Have (Sprint 5--6):} Version history, export, sharing UI, presence polish, accessibility.
\item \textbf{Could Have (Sprint 7+):} Admin dashboard, org-level settings, advanced AI features (restructure, full-document analysis).
\end{specenum}

\subsection{Infrastructure \& Non-Feature Work}

Non-feature work (database setup, CI pipeline, testing infrastructure, Docker Compose) is explicitly scheduled in Sprint~1 as ``infrastructure stories'' with the same acceptance criteria rigor as feature stories. Example: ``CI pipeline runs linting, type checking, and unit tests on every PR push, verified by a passing green check on a test PR.'' This prevents the common antipattern of perpetually deferred infrastructure.

  \textbf{M2: Core CRUD + Auth.}
  Document CRUD API (\texttt{POST/GET/PATCH/DELETE /api/documents}). JWT auth (login, register, refresh). Frontend connects to API: create document, load document. &
  (1) Integration tests pass for all document endpoints.
  (2) Unauthenticated requests return 401.
  (3) Frontend creates and loads a document via the API. \\
\addlinespace[4pt]
\textbf{Sprint 3} & Mar 23--26 &
  \textbf{M3: Real-Time Sync.}
  Hocuspocus server with Yjs. WebSocket connection from frontend. Two browser tabs sync edits in real-time. Presence (cursors) displayed. &
  (1) Two tabs editing the same document see each other's changes within 300\,ms.
  (2) Cursor positions are visible across tabs.
  (3) Disconnecting and reconnecting restores state. \\
\addlinespace[4pt]
\textbf{Sprint 4} & Mar 27--30 &
  \textbf{M4: AI Integration.}
  AI invocation endpoint. Job queue (BullMQ). LLM provider adapter (one provider). Proposal UX (diff overlay, accept/reject). Token metering. &
  (1) User selects text and invokes ``Rewrite''; AI proposal appears as a diff.
  (2) Accept commits to CRDT; reject dismisses.
  (3) Token count is logged per request. \\
\addlinespace[4pt]
\textbf{Sprint 5} & Mar 31--Apr 2 &
  \textbf{M5: Permissions + Sharing.}
  Role-based access control (Owner/Editor/Commenter/Viewer). Share document by email. Permission enforcement on API and WebSocket. &
  (1) Viewer cannot edit. Commenter can comment but not edit.
  (2) Sharing by email sends invitation.
  (3) Permission revocation terminates active WebSocket within 60\,s. \\
\addlinespace[4pt]
\textbf{Sprint 6} & Apr 3--5 &
  \textbf{M6: Polish + Hardening.}
  Version history UI. Export (PDF/DOCX). Offline editing. Presence collapse ($>$10 users). Accessibility pass. Load testing. &
  (1) Version history shows $\ge$5 snapshots for an actively edited document.
  (2) Export produces valid DOCX.
  (3) k6 load test: 50 concurrent editors, p95 $\le$150\,ms. \\
\bottomrule
\end{tabularx}
}

\vspace{8pt}
\begin{notebox}[Dependency Chain]
\small M1 (skeleton) $\to$ M2 (CRUD+Auth) $\to$ M3 (real-time sync) $\to$ M4 (AI integration) $\to$ M5 (permissions) $\to$ M6 (polish). Each milestone builds on the previous. The AI integration (M4) deliberately comes \emph{after} real-time sync (M3) because the AI proposal conflict resolution depends on a working CRDT layer.
\end{notebox}
